TGR files

Version 1

Status: Implemented

The file starts with a 32 bit version number as header. Following that are the content blocks Every block starts with a 32bit block header, this header can either be a block length or just additional information as well as UINT32_MAX (0xFFFFFFFF) which markes the content types end. This will change content type in the following order

  • Textures

    The header is the size of the texture file in byte. The texture file is a standard PNG file.

  • Materials

    The header is the size of the following Material in byte. Currently this should either be 20 or 24. The content consists of the Material data.

  • Actors

    The header consists of the size of all the vertex data in the actor (vertexCount * vertexStride). The content starts with the properties of the Actor. Following is the 4 byte indexCount, an other 4 byte for vertexCount, indexCount times 4 byte index values. Lastly the vertex data is appanded

Note: All values are unchecked by the reader, it assumes that data is transfered and baked correctly, this might change in further versions

Version 2

Status: RFC

The file starts with a 32 bit version number as header. Following that are the content blocks. Every block starts with a 32bit block header, this header can either be a block length or just additional information as well as UINT32_MAX (0xFFFFFFFF) which markes the content types end. Each content type starts with the 32bit count of content blocks within this type. Optional content types can be decleared empty with the count being zero followed by the content type end marker. This will change content type in the following order:

  • Textures

    The header is the size of the texture file in byte. The texture file is a standart PNG file.

  • Materials

    The header is the size of the following Material in byte. Currently this should either be 20 or 24. The content consists of the Material data.

  • Actors

    The header consists of the size of all the vertex data in the actor (vertexCount * vertexStride). The content starts with the properties of the Actor. Following is the 4 byte indexCount, an other 4 byte for vertexCount, indexCount times 4 byte index values. Lastly the vertex data is appanded

Optional

  • Animation
  • Font

    The header is the size of the font file in byte. The font file is a standard true type font file.

  • UI Boundingboxes

    The header is reserved for further use and should be 0. The content consists of 8 times 32bit float values indicating 4 Points in a 2D plain, representing the actual Hitbox of a UI Actor. The coordintaes should be homogeneous within a range of -1 to 1

  • Sampler

    The header is reserverd for further use. The content consists of a SamplerInputInfo.

Note: All values are unchecked by the reader, it assumes that data is transfered and baked correctly, this might change in further versions
